====================================
1. צמצום הרווח הגדול בפוסטר
====================================

בעיה:
בפוסטר יש רווח גדול מדי בין:
שם המיזם / תיאור קצר של המיזם
לבין תחילת אזור התשובות והתיבות.

ב-config.js כרגע:
projectName נמצא באזור העליון
description נמצא מתחתיו
אבל התיבות הראשונות מתחילות נמוך מדי, למשל problem מתחיל סביב y=720 בפוסטר לאורך.

יש לצמצם את הרווח האנכי כך שהפוסטר ירגיש מלא ומאוזן יותר.

דרישה:
- לקרב את אזור התיבות למעלה.
- לצמצם את הפער בין description לבין problem / requirements / תחילת התוכן.
- לא לפגוע בלוגו, בכותרת ובקריאות.
- לבדוק גם A4 לאורך וגם A4 לרוחב.

הנחיה:
ב-poster-builder/src/products/physical/config.js
לעדכן את PORTRAIT_RECTS ו-LANDSCAPE_RECTS כך שהשדות הראשונים של התוכן יתחילו גבוה יותר.

לדוגמה:
ב-A4 לאורך, אם description מסתיים בערך ב-y=430, אין צורך להתחיל את התוכן רק ב-y=720.
יש להוריד את תחילת התוכן לאזור קרוב יותר, למשל סביב y=600 או פחות, בהתאם לעיצוב.
בהתאם לכך יש להזיז מעלה גם את שאר השדות ולשמור על מרווחים תקינים.

====================================
2. התאמת גודל תיבות בממשק לשדות קצרים עד 80 תווים
====================================

בעיה:
בממשק מילוי השאלות, שדות של עד 80 תווים מקבלים תיבה גדולה מדי.
לדוגמה שדות קצרים כמו צבעים, שם קצר, תיאור קצר או תשובות מוגבלות עד 80 תווים לא צריכים להיראות כמו תיבת טקסט גדולה.

דרישה:
- אם maxLength או maxChars הוא עד 80 תווים, התיבה צריכה להיות נמוכה וקומפקטית יותר.
- שדות קצרים יוצגו כ-input או textarea בגובה קטן.
- שדות ארוכים יותר יישארו textarea רגיל.
- לא לפגוע במונה התווים.
- לא לפגוע בשדות של 110/130 תווים או רשימות.

הנחיה:
ב-split-flow/main.js או ברכיב/פונקציית הרינדור של השדות:
להוסיף class לפי אורך השדה, למשל:
short-field עבור max <= 80
medium-field עבור max <= 110
long-field עבור מעל 110

וב-CSS:
.short-field input,
.short-field textarea {
  min-height: 44px;
  height: 44px;
}

.medium-field textarea {
  min-height: 70px;
}

.long-field textarea {
  min-height: 100px;
}

אם יש textarea לשדה קצר, להגדיר rows=1 או rows=2 בלבד.

====================================
3. מידות תמונות לפי המידות הפיזיות של הפוסטר
====================================

בעיה:
הפרומפט והתמונה לא תמיד בנויים לפי מידות הסלוט בפוסטר.
כרגע app ו-website מקבלים 3 סלוטים דומים, אבל מסך אפליקציה לא יכול להיות ריבועי או כמעט ריבועי.
צריך שהמידות של הסלוט בפוסטר והפרומפט יהיו מתואמים מראש.

דרישה:
- לכל סוג מוצר תהיה גיאומטריית תמונות מתאימה:
  app = מסכים אנכיים ביחס 9:16
  website = מסכי אתר ביחס אופקי, למשל 16:9 או יחס קרוב לסלוט
  physical = להשאיר את הלוגיקה הקיימת של תמונת מוצר / תמונת שימוש

- לא להציג מסכי אפליקציה בסלוט ריבועי או כמעט ריבועי.
- הפרומפט צריך לבקש בדיוק יחס שמתאים לסלוט בפוסטר.
- הפוסטר צריך להציג את התמונות ביחס הנכון, בלי עיוות ובלי חיתוך בעייתי.

הנחיה:
ב-poster-builder/src/products/physical/config.js
לעדכן את getVisualSlots כך שיהיה בידול בין app לבין website.

עבור productType === 'app':
- ליצור 3 סלוטים אנכיים ביחס 9:16.
- למשל ב-A4 לאורך:
  גובה הסלוט יכול להישאר באזור 420–520px, אבל הרוחב צריך להיגזר לפי 9:16:
  width = height * 9 / 16
- למקם את שלושת המסכים באופן מאוזן בתוך אזור התמונות.
- לא להשתמש ברוחב שמייצר יחס כמעט ריבועי.
- לשמור gap קבוע בין הסלוטים.
- לוודא שהם נכנסים בתוך שטח הפוסטר ולא מתנגשים בשדות הטקסט.

עבור productType === 'website':
- להשתמש בסלוטים אופקיים ביחס שמתאים למסך אתר, למשל 16:9 או 4:3.
- לא להשתמש באותו מבנה כמו app אם הוא גורם לעיוות.

עבור productType === 'physical':
- לא לשנות את המבנה הקיים בלי צורך.

4. התאמת הפרומפט למידות הסלוט בפוסטר
====================================

בעיה:
גם אם הסלוט בפוסטר תוקן, הפרומפט חייב לבקש מראש תמונה באותו יחס.
אחרת המערכת תייצר תמונה שלא מתאימה לסלוט ותיווצר חיתוך/עיוות.

דרישה:
ב-split-flow/main.js, במחולל הפרומפטים של app/website:
- להשתמש ביחס הפיזי של הסלוטים מתוך getVisualSlots.
- עבור app להוסיף בבירור:
  portrait app screen, 9:16 ratio, screen UI only
- עבור website להוסיף:
  website screen UI, wide screen ratio, 16:9 or according to poster slot
- לא לבקש תמונה ריבועית לאפליקציה.
- לא לבקש טלפון, רק מסך עצמו.

הטקסט שצריך להיכנס לכל פרומפט app:
Create only the app screen UI itself, in a vertical 9:16 portrait ratio.
The screen must fill the entire image edge-to-edge.
Do not include a phone frame, device mockup, hands, desk, room, background, decorative scene, or objects around the screen.
Generate a lightweight web-optimized UI mockup suitable for placing directly inside the poster slot.

הטקסט שצריך להיכנס לכל פרומפט website:
Create only the website screen UI itself, in a wide website screen ratio matching the poster slot.
The screen must fill the entire image edge-to-edge.
Do not include a laptop frame, browser chrome, desk, room, people, background scene, or objects around the screen unless explicitly required.
Generate a lightweight web-optimized UI mockup suitable for placing directly inside the poster slot.

====================================
5. כותרות חסרות בפוסטר: הערך / המשוב / מה שיפרנו
====================================

בעיה:
בפוסטר לא מופיעות הכותרות:
- מה הערך המרכזי של הפתרון?
- המשוב שקיבלנו
- מה שיפרנו

בדיקה בקוד:
ב-config.js השדות האלה מוגדרים עם:
noLabel: true

וב-editor.js הכותרת נוצרת רק אם !field.noLabel.
לכן הכותרות לא מופיעות.

דרישה:
הכותרות חייבות להופיע בפוסטר.

תיקון:
ב-poster-builder/src/products/physical/config.js
לשנות את השדות:
value
feedbackReceived
improvementsAfterFeedback

כך שלא יהיו noLabel: true.

כלומר:
- להסיר noLabel: true
או להגדיר noLabel: false

בנוסף, בגלל שהשדות האלה קטנים יחסית, צריך להגדיר להם titleSpacing קטן יותר, למשל:
titleSpacing: 42
או 48

ולהקטין מעט fontSize אם צריך, למשל:
fontSize: 32–36
minFontSize: 22–24

כדי שגם הכותרת וגם התשובה ייכנסו באותה תיבה.

חשוב:
לא מספיק רק להסיר noLabel.
צריך לוודא שהתיבות עצמן מספיק גבוהות כדי להכיל גם כותרת וגם תוכן.

ב-PORTRAIT_RECTS:
להגדיל מעט את height של value / feedbackReceived / improvementsAfterFeedback
או להזיז אותם כך שלא ידרסו אחד את השני.

ב-LANDSCAPE_RECTS:
אותו דבר — לוודא שיש מספיק גובה לכותרת + תשובה.

====================================
6. רינדור תמונות בפוסטר ללא עיוות
====================================

בעיה אפשרית:
ב-editor.js הפונקציה applySlotImage משתמשת ב:
Math.max(slot.width / img.width, slot.height / img.height)

זה גורם לתמונה למלא את כל הסלוט אבל עלול לחתוך קצוות אם היחס לא מתאים.
אחרי שמתקנים את יחס הסלוט והפרומפט זה אמור להשתפר, אבל עדיין צריך לבדוק.

דרישה:
- אם התמונה והסלוט באותו יחס, לא אמור להיות חיתוך.
- עבור app, אם הסלוט 9:16 והפרומפט 9:16, התמונה צריכה להיכנס נקי.
- עבור website, אם הסלוט wide והפרומפט wide, התמונה צריכה להיכנס נקי.

אם עדיין יש חיתוך משמעותי, לשקול מעבר מ-cover ל-contain עבור app/website בלבד:
scale = Math.min(slot.width / img.width, slot.height / img.height)

אבל רק אם זה לא משאיר שוליים לא רצויים.
עדיף קודם לתקן יחס סלוט + יחס פרומפט.
